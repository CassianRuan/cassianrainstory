# 找不同网页游戏 MVP 分阶段执行方案

## 1. 项目目标

在 3 天内上线一个可被公网用户直接访问的桌面端中文“找不同”网页游戏，完成以下核心闭环：

1. 玩家用邮箱、密码和昵称注册，注册后立即登录游玩。
2. 玩家按顺序解锁 20 个关卡。
3. 每关左右并排展示两张图；左图仅供观察，玩家只能点击右图。
4. 每关有独立的不同点数量、限时和 3 次误点机会。
5. 玩家通关后解锁下一关；失败后可立即重试。
6. 每关排行榜先按玩家首次通关前的累计挑战次数升序，再按首次通关用时升序。
7. 通关后可以重复游玩，但不改变该关排名。
8. 管理员可上传三张关卡图片、自动提取红框答案、配置关卡并清除异常成绩。

首发目标规模为同时在线不超过 20 人。第一版仅适配电脑浏览器、仅支持简体中文、完全免费。

## 2. 已确认需求与规则

### 2.1 玩家端

- 账号：邮箱 + 密码 + 昵称。
- 注册后直接可玩，不验证邮箱。
- 支持登录、退出、修改昵称。
- 第一版不做找回密码、头像、修改密码、第三方登录。
- 关卡必须按配置顺序依次解锁。
- 所有玩家都能查看每一关排行榜。
- 左右图并排，左图不可点击且不显示命中标记。
- 右图可点击；命中后仅在右图显示已找到标记。
- 每关错误机会固定为 3 次；用尽或超时即失败，可立即重试。
- 不同点全部找到即通关。
- 玩家通关后仍能重复游戏，但后续成绩不更新首次通关排行榜记录。

### 2.2 排名口径

- “挑战次数”指该玩家从第一次正式开始该关，到第一次成功通关为止的累计次数。
- 点击“开始挑战”且关卡图片加载完毕后，由服务端创建一次挑战，此时计数；刷新、退出或放弃也算一次，防止通过中断失败局规避计数。
- 首次通关后，将累计挑战次数和本次通关用时固化为排名成绩。
- 排序字段依次为：`首次通关累计挑战次数 ASC`、`首次通关用时 ASC`、`首次通关时间 ASC`。
- 同一玩家在每关只有一条排名记录。
- 管理员清除异常成绩后，该成绩不再出现在排行榜；玩家的通关与解锁状态默认保留。若需要允许该玩家重新上榜，管理员可选择“清除并重置该关首次通关记录”。

### 2.3 关卡素材

每关由管理员提供：

1. 左侧原图；
2. 右侧差异图；
3. 与右图尺寸完全一致的红框标注图。

标注图约束：每个不同点使用一个纯红色矩形框，红框不重叠。后台在浏览器中自动识别红色像素、合并矩形边界并生成归一化点击区域，不把标注图公开给玩家。管理员必须在预览界面逐个检查自动识别结果后才能发布。

## 3. 技术方案

### 3.1 技术选型

| 层级 | 方案 | 选择原因 |
| --- | --- | --- |
| 前端 | React + TypeScript + Vite | 满足 React 要求，开发和构建速度快 |
| UI | Tailwind CSS + 少量自有组件 | 三天内快速完成统一桌面界面 |
| 路由/请求 | React Router + TanStack Query | 页面和服务端状态处理成熟 |
| 后端 | Supabase（Auth、PostgreSQL、Storage、RPC） | 免自建服务器，账号、数据库和图片存储一体化 |
| 部署 | Vercel 托管前端，Supabase 托管后端 | 自动 HTTPS、公网访问和部署简单 |
| 自动检查 | Vitest + React Testing Library | 覆盖核心规则和交互 |

部署完成后，用户通过 Vercel 公网 HTTPS 地址访问；后续可绑定自有域名。

### 3.2 安全和防作弊基线

- 数据库启用 Row Level Security；普通玩家不能写关卡、答案区域或排行榜。
- 答案区域不得随关卡列表接口下发，只在服务端判定玩家点击是否命中。
- 开始挑战、提交点击、完成挑战通过数据库 RPC 执行，服务端时间为准。
- 每个挑战生成唯一会话，服务端记录开始时间、错误次数和已命中区域。
- 服务端验证关卡限时、点击坐标、重复命中和挑战状态。
- 管理功能仅限 `profiles.role = admin` 的账号。
- 图片限制 MIME 类型、大小和尺寸；文件名使用系统生成 ID。
- 三天版不承诺抵抗专业攻击者；异常检测、验证码和设备指纹列入后续增强。

## 4. 功能范围与页面

### 4.1 玩家页面

- `/register`：邮箱、密码、昵称注册。
- `/login`：邮箱、密码登录。
- `/levels`：关卡列表，显示已通关、已解锁、未解锁状态。
- `/play/:levelId`：左右图片、倒计时、已找到数量、剩余错误机会、成功/失败弹窗。
- `/leaderboard/:levelId`：关卡排行榜，显示名次、昵称、累计挑战次数、首次通关用时。
- `/profile`：查看邮箱、修改昵称、退出。

### 4.2 管理页面

- `/admin/levels`：关卡配置表、排序、显示/隐藏、创建和编辑。
- `/admin/levels/:id/assets`：上传三图、自动提取红框、答案区域预览校验。
- `/admin/leaderboards`：按关卡查看成绩、搜索昵称/邮箱、清除异常成绩。

首个管理员账号通过 Supabase 控制台创建，并在数据库中赋予 `admin` 角色，不开放网页端管理员注册。

## 5. 关卡配置表

后台采用可编辑表格，保存时统一校验：

| 字段 | 示例 | 规则 |
| --- | --- | --- |
| 关卡编号 | `level-01` | 系统生成，唯一 |
| 关卡名称 | `咖啡馆` | 必填，1–40 字 |
| 显示顺序 | `1` | 正整数，不重复 |
| 是否显示 | `是` | 隐藏后玩家不可新开挑战 |
| 限时时长（秒） | `90` | 建议 15–600 |
| 不同点数量 | `5` | 必须等于识别出的有效红框数 |
| 错误机会 | `3` | MVP 固定为 3，仅展示不可编辑 |
| 左图 | 文件 | JPG/PNG/WebP，建议同一比例 |
| 右图 | 文件 | 必须与标注图尺寸一致 |
| 红框标注图 | 文件 | 仅管理端可访问 |
| 发布状态 | 草稿/已发布 | 素材和配置校验通过后才能发布 |

顺序调整后，已解锁状态按新的关卡顺序重新计算可能引发歧义。MVP 约束为：正式上线后只在末尾新增关卡，不重排已发布关卡；确需调整时由管理员确认风险。

## 6. 核心数据模型

| 表 | 关键字段 | 用途 |
| --- | --- | --- |
| `profiles` | `user_id, nickname, role, created_at` | 用户资料和管理员角色 |
| `levels` | `id, title, sort_order, visible, status, time_limit_sec, difference_count, image paths` | 关卡配置 |
| `difference_regions` | `id, level_id, x, y, width, height` | 归一化答案区域，仅服务端/管理员可读 |
| `challenge_runs` | `id, user_id, level_id, started_at, finished_at, status, wrong_count` | 每次挑战记录 |
| `challenge_hits` | `run_id, region_id, hit_at` | 当前挑战已找到的不同点 |
| `level_progress` | `user_id, level_id, attempt_count, first_cleared_at, first_clear_ms` | 首次通关及固定排名数据 |
| `score_audit_log` | `id, operator_id, target_user_id, level_id, action, reason, created_at` | 异常成绩清理审计 |

关键唯一约束：`level_progress(user_id, level_id)`、`challenge_hits(run_id, region_id)`、`levels(sort_order)`。

## 7. 服务端操作边界

- `start_challenge(level_id)`：验证登录、可见性和顺序解锁；累加首次通关前的挑战次数；创建挑战。
- `submit_click(run_id, normalized_x, normalized_y)`：验证时限和会话；返回命中、重复命中、误点、失败或完成状态。
- `finish_challenge(run_id)`：由最后一次有效命中触发；首次通关时原子写入固定成绩并解锁下一关。
- `get_level_leaderboard(level_id, page)`：只返回公开字段，按既定规则分页排序。
- 管理 RPC：保存关卡、发布关卡、清除/重置异常成绩，全部验证管理员角色并写审计日志。

坐标统一存为相对图片宽高的 `0–1` 小数，避免浏览器缩放导致命中区域偏移。点击区域可在红框边界外增加少量容错（建议图片宽高的 0.5%），后台预览时确认。

## 8. 三天执行计划

### 阶段 0：启动与素材检查（第 1 天，约 1 小时）

- 初始化 React、Supabase 和 Vercel 项目及环境变量样例。
- 确认至少 2 个真实关卡素材可用于端到端验证。
- 建立数据库迁移、基础目录、错误处理和部署流水线。

验收：空应用可从公网测试地址访问，数据库迁移可重复执行。

### 阶段 1：账号、数据和关卡骨架（第 1 天，约 5 小时）

- 完成注册、登录、退出、昵称修改和路由保护。
- 建表、RLS、管理员角色和 Storage 策略。
- 完成关卡列表及顺序解锁查询。
- 完成后台关卡配置表和显示/顺序设置。

验收：普通用户只能访问自己的资料；管理员可新建草稿关卡；锁定关卡不可进入。

### 阶段 2：核心游戏闭环（第 1–2 天，约 7 小时）

- 实现三图上传、红框自动识别、预览与发布校验。
- 实现双图布局、右图坐标换算、倒计时、命中标记和 3 次错误机会。
- 实现服务端挑战会话、命中判定、失败重试、首次通关和解锁。
- 补充核心单元测试。

验收：成功、超时、三次误点、刷新中断、重复点击等路径结果正确；左图无法点击且无标记。

### 阶段 3：排行榜与后台治理（第 2 天，约 5 小时）

- 实现首次通关成绩固化及复玩不改排名。
- 实现每关排行榜和分页。
- 实现异常成绩清除/重置与审计日志。
- 导入并配置 20 个正式关卡。

验收：排行榜严格按挑战次数、用时、通关时间排序；重复游玩不改变成绩；普通用户无法调用管理操作。

### 阶段 4：联调、验收与发布（第 3 天，约 6 小时）

- 在 Chrome、Edge 最新版测试 1440×900 和 1920×1080 桌面分辨率。
- 执行核心自动测试、数据库权限测试和人工冒烟测试。
- 检查 20 关的素材尺寸、红框区域、时间和不同点数量。
- 配置生产环境、创建管理员、部署并执行发布后冒烟测试。
- 输出管理员操作说明和已知限制。

验收：新用户可从注册连续完成首关、进入次关并查看正确排行榜；管理员能独立上传和发布新关卡。

> 工期估算基于素材按约定格式及时齐备、无需大幅视觉返工。20 关素材检查与配置是 3 天目标中最容易波动的部分，应尽早提供。

## 9. MVP 验收清单

- [ ] 公网 HTTPS 地址可访问，未登录用户被引导登录。
- [ ] 注册后直接登录；重复邮箱、弱密码和重复昵称策略有明确提示。
- [ ] 只能按顺序解锁可见关卡。
- [ ] 左图不可点击，命中标记只出现在右图。
- [ ] 倒计时由服务端时间校验，3 次误点后失败并可重试。
- [ ] 不同点全部找到后只结算一次。
- [ ] 首次通关累计挑战次数准确，刷新/放弃不能规避计数。
- [ ] 排行榜排序正确，通关后复玩不改变排名。
- [ ] 玩家无法读取答案区域或写入排行榜。
- [ ] 管理员能完成上传、识别、预览、配置、发布和异常成绩清理。
- [ ] 20 个关卡逐关通过人工点位验证。
- [ ] Chrome 和 Edge 最新版无阻断性问题。

## 10. 明确不在第一版范围内

- 手机和平板适配、多语言、社交登录、邮箱验证、找回/修改密码。
- 头像、个人主页、好友、分享、评论、成就、道具、广告和付费。
- 音效、复杂动画、实时多人竞技、全站综合排行榜。
- 高强度反作弊、自动异常检测、内容审核和封禁系统。
- 高可用多区域部署及大规模压力优化。

## 11. 风险与应对

| 风险 | 影响 | MVP 应对 |
| --- | --- | --- |
| 20 关素材到达较晚或标注不规范 | 阻塞最终验收 | 第一天先验 2 关；后台强制尺寸和红框数校验 |
| 客户端计时或答案泄露导致作弊 | 排行榜失真 | 服务端计时和判定，答案区域不下发 |
| 网络延迟影响用时 | 小幅排名误差 | 用服务端开始/完成时间统一口径；低并发场景可接受 |
| 注册不验证邮箱 | 垃圾账号或邮箱冒用 | MVP 接受；后续增加邮箱验证和限流 |
| 免费托管额度或平台限制变化 | 部署受限 | 上线前核对当前额度；数据模型保留迁移能力 |
| 管理员误清成绩 | 用户争议 | 二次确认、必填原因、审计日志；默认不删除挑战原始记录 |

## 12. 后续迭代建议

1. 邮箱验证、找回密码、限流和验证码。
2. 可配置排行榜规则与赛季榜。
3. 异常成绩自动标记、挑战回放和管理审核。
4. 音效、提示道具、每日挑战和成就系统。
5. 移动端适配、CDN 图片优化和更高并发压测。

## 13. 开工前输入物

- 20 关的左图、右图和红框标注图，按关卡编号整理。
- 每关名称、显示顺序、限时秒数、不同点数量和是否显示。
- 用于创建 Supabase、Vercel 和首个管理员的项目方账号信息（凭据不写入仓库）。
- 如需自有域名，在发布前提供域名控制台权限；否则先使用平台默认公网域名。
